1.全面概述

Apache Maven 是一种软件项目管理和理解工具。基于项目对象模型的概念（POM），Maven 可以从中央信息中管理项目的构建，报告和文档。

2.基本功能

因此实际上 Maven 的基本功能就是作为 Java 的项目包管理器，可以快速配置项目的依赖并且进行版本配置管理，其配置的理念是基于项目对象模型 project object model。具体来说，Maven 回使用 pom.xml 这种项目描述文档来描述一个项目以及对应的依赖。

然后再基于这份文档，使用 maven 的命令行工具进行依赖管理即可，类似 Cpp 的 Vcpkg 工具、Python 的 Pip 工具...

3.快速使用

3.2.可选配置

  • MAVEN_ARGS 环境变量(操作 MVC 的参数) 从 Maven 3.9.0 开始，此变量包含传递给 Maven 的参数，优先于用户的命令行参数。例如，可以通过 -B -V checkstyle:checkstyle 定义选项和目标。其中 -B 表示跳过交互式提示，-V 表示显示 Maven 的版本信息，checkstyle:checkstyle 是一个目标任务，表示运行 Checkstyle 插件进行代码检查。

  • MAVEN_OPTS 环境变量(操作 JVM 的参数) 该变量包含用于启动 Maven 所需的 JVM 参数，可以用来向 JVM 提供额外的选项。例如，JVM 的内存设置可以通过 -Xms256m -Xmx512m 来定义。

  • settings.xml 文件 settings.xml 用于存储跨项目的 Maven 配置，通常存储在两个目录中

    • 用户级别配置目录 $USER_HOME/.m2/settings.xml
    • 全局级别配置目录 $MAVEN_HOME/conf/settings.xml

    您可以稍微查看一下全局级别配置文件，稍微了解一下，当存在用户级别配置文件时，就会优先使用用户级别配置文件（通常进入那些大型公司会要求替换这个 settings.xml 文件，因为这些公司通常都会有自己的依赖链路要求，同时这个文件一般还会指定一个用于存储从远程仓库拉取到本地的依赖文件存储目录）。

  • .mvn 目录 位于 Maven 项目的顶级目录中，一般包含三个文件

    • .mvn/maven.config 从 Maven 3.3.1+ 开始，可以通过 .mvn/maven.config 文件简化 Maven 命令行参数的设置（每个参数最好直接单独一行），这样就不再需要使用 MAVEN_ARGS

    • .mvn/jvm.config 从 Maven 3.3.1+ 开始，可以通过 .mvn/jvm.config 文件为每个项目定义 JVM 配置（每个参数最好直接单独一行），这样就不再需要使用 MAVEN_OPTS

    • .mvn/extensions.xml 过去的做法（直到 Maven 3.2.5）是创建一个 JAR 文件并将其手动放入 ${MAVEN_HOME}/lib/ext 目录中，或者通过 mvn -Dmaven.ext.class.path=extension.jar 命令行选项来加载扩展。现在可以通过定义 .mvn/extensions.xml，一般实际开发中用来拓展 Maven 核心功能的，和 settings.xml 作用还是有很大不同的，不要混淆一谈，有需要了解一下即可（修改后可用 mvn help:effective-settings 进行配置检查）。

重要

补充：不过一般情况下，我们无需改动以上的环境变量和配置文件。

  • 对于环境变量一般使用 .mvn/maven.config 和 .mvn/jvm.config 来替代，这样每一个项目配置起来比较灵活
  • 一般也用不到 .mvn/extensions.xml 这种需要拓展 Maven 功能。
  • 而对于经过正常魔法手段的网络环境下，一般也无需配置 settings.xml，使用默认的即可，如果需要修改，一般也只有以下三个个人开发者需求（最好只修改用户级别的 settings.xml 配置文件）
    • 可能需要改动依赖存储的目录，以避免系统磁盘占用过多，只需要改动 <localRepository> 标签即可
    • 可能需要改动网络代理的配置，以使用魔法阵来健康上网，只需要改动 <proxy> 标签即可
    • 可能需要配置一些插件，后续使用 Maven 管理的 Java 项目（这里指专门开发 Maven 插件的 Java 项目）开发完毕后进行 mvn install 后会把插件安装到本地仓库 ~/.m2/repository/ 中，此时需要使用这里面的插件标签来标识该插件以简化命令行的使用

因此总结来说，使用 .mvn/maven.config、.mvn/jvm.config、settings.xml 的概率是最大的。

另外，一般开发环境使用 mvn 的命令行多一些，因为命令行的可选参数优先级最高，可以覆盖 .mvn/maven.config、.mvn/jvm.config 这两个配置文件。而测试环境、生产环境一般使用 .mvn/maven.config、.mvn/jvm.config 这两个配置文件会多一些，因为更加方便。

3.3.常用指令

这里先列出并且大概说下 mvn 工具的常见指令。

  • mvn [options] [<goal(s)>] [<phase(s)>] 使用通用的
    • options 可选参数，比如 -Dproperty=value（设置属性）、-X（启用调试模式）等
    • goal 具体目标，通常是指执行某些自定插件
    • phase 生命阶段，通常是指生命周期内的阶段
  • mvn -h 可以查询相关的指令帮助
  • mvn verify 最为典型的指令调用

重要

补充：Maven 生命周期是一套预定义的 phases, 构建阶段，确保项目按照既定顺序进行构建。Maven 有 3 个 Lifecycle, 生命周期，分别是

  • clean（“清理” 生命周期），包含多个阶段。本生命周期内，在执行某个阶段时，会自动执行该阶段前面的所有阶段
    • pre-clean：执行清理前的操作
    • clean：删除 target/ 目录
    • post-clean：执行清理后的操作
  • default（“构建” 生命周期），包含多个阶段。本生命周期内，在执行某个阶段时，会自动执行该阶段前面的所有阶段
    • validate 验证项目是否正确，并检查 pom.xml 配置
    • compile 编译源码（src/main/java）到 target/classes
    • test 运行单元测试（基于 src/test/java），不过有时可以使用 mvn clean package -DskipTests 跳过测试阶段
    • package 将编译后的代码打包（如 jar、war）
    • verify 运行集成测试，确保打包的应用正常
    • install 安装 jar/war 到本地仓库下（~/.m2/repository/），一般会安装一些项目，比如自定义的插件项目和其他的服务项目，以方便别的项目进行本地依赖
    • deploy 发布 jar/war 到远程仓库（如 Nexus）
  • site（“文档” 生命周期），包含多个阶段。本生命周期内，在执行某个阶段时，会自动执行该阶段前面的所有阶段
    • pre-site：执行文档生成前的任务
    • site：生成 HTML 文档
    • post-site：执行文档生成后的任务
    • site-deploy：发布站点到服务器

而在 pom.xml 中的 plugin, 插件 标签可以绑定到特定阶段，扩展构建过程。也就是说，您可以在 Maven 的构建过程中，插入额外的任务（例如代码生成、代码检查、打包方式自定义等），以干预生命周期的行为。

package com.work;

import com.google.gson.Gson;

public class App {
    public static void main(String[] args) {
        // 创建一个对象
        Person person = new Person("John", 30);

        // 创建Gson实例
        Gson gson = new Gson();

        // 将对象转换为 JSON 字符串
        String json = gson.toJson(person);
        System.out.println("JSON Output: " + json);

        // 将 JSON 字符串转换为对象
        Person personFromJson = gson.fromJson(json, Person.class);
        System.out.println("Person Name: " + personFromJson.getName());
        System.out.println("Person Age: " + personFromJson.getAge());
    }
}

class Person {
    private String name;
    private int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public String getName() {
        return name;
    }

    public int getAge() {
        return age;
    }
}

  • -source 17 设置 .java 源码版本
  • -target 17 设置 .class 目标版本

自动管理 Java 版本的 API 可用性，如 Java 17 移除的 API 在低版本 JVM 运行时将报错。

4.深入学习

4.1.父子模块

Maven 多模块项目中，父 POM 的 <modules> 是 “子模块的清单”，通常需要写明父 POM 的打包类型为<packaging>pom</packaging>（父 POM 的基本要求），而核心价值是：生命周期联动，父 POM 的构建指令（编译、测试、打包、部署等）会自动传递到所有子模块，实现 “一键发布所有子模块”。

4.2.继承依赖

  • 而 父子版本统一 则需要子模块使用 <parent> 结合父 POM 的 <dependencyManagement>，所有子模块可复用统一的版本规则，避免版本不一致。并且可以在 <parent> 内部使用 <relativePath/> 来控制查找父 POM 文件的查找顺序。
  • 但是如果父 POM 直接用 <dependencies> 引入依赖（没写<dependencyManagement>），子 POM 继承父 POM 后，依赖版本和传递逻辑会如何变化呢？结论是：子 POM 会继承父 POM 中<dependencies> 里的依赖（含版本）。
写法XML 本质Maven 查找顺序
不配置 <relativePath>默认值 ../pom.xml本地目录（本地上级目录 pom.xml ） → 本地仓库 → 远程仓库
<relativePath/>空字符串 ""（跳过本地路径）本地仓库 → 远程仓库
<relativePath>../parent/pom.xml</relativePath>具体路径指定本地路径 → 本地仓库 → 远程仓库

重要

补充：Maven 的 <parent> 是单继承的，那么一个子模块只能声明一个 <parent> 父 POM，是无法同时继承多个 POM 的配置。

但实际开发中，你往往需要：

  1. 继承公司内部的父 POM（包含公司统一的编码、插件、私服配置）
  2. 同时复用 Spring Boot 的 BOM（统一 Spring Boot 依赖版本）
  3. 还可能需要复用 Spring Cloud 的 BOM（统一微服务依赖版本）

这个时候使用 <dependency> 来引入依赖的时候，就不仅仅要求引入 .jar 包了，还需要 <type>pom</type> 来引入其他第三方清单的配置（并且可以搭配后续介绍的 <scope>import</scope> 标签搭配使用）。

4.2.仲裁版本

先来看 Maven 在引入相同但版本不同的依赖时的行为。

优先级规则名称核心逻辑
1直接声明优先若你在项目 POM 中直接显式声明了依赖的版本，无论传递依赖是什么版本，都用你声明的
2最短路径优先依赖在 “依赖树” 中层级越浅，优先级越高（直接依赖 > 一级传递依赖 > 二级传递依赖）
3声明顺序优先若路径长度相同，Maven 会选择 POM 中先声明的那个依赖所带的版本
4依赖管理优先若父 POM / 当前 POM 的 <dependencyManagement> 中声明了版本，会覆盖上述规则

而上述的规则也被叫做 依赖仲裁, Dependency Mediation，也常被简称为 版本仲裁，核心是指当项目中多个依赖间接引入了同一个组件的不同版本时，Maven/Gradle 会按照既定规则“裁定”最终使用哪个版本（但是开发者也可以手动干预版本仲裁，强制让项目使用某个版本）。

不过我们举个实际场景：你的项目直接依赖 A 组件（A 依赖 snakeyaml 1.29），同时依赖 B 组件（B 依赖 snakeyaml 1.30），此时 Maven 会默认按「最短路径」「声明顺序」等规则自动选一个版本（比如 1.30）。但如果 1.29/1.30 有漏洞，你需要强制用安全的 1.33 版本，就需要手动仲裁。其中“手动仲裁版本”有两种常见方式（以 Maven 为例）：

  1. 最简单的方式：直接声明依赖（简单直接），在 Maven 中「直接声明的依赖版本」优先级高于「间接依赖的版本」，相当于你手动“拍板”用 1.33，不管其他依赖引入的是哪个版本。适用场景，快速解决单个组件的版本问题，适合临时调整。
  2. 更规范的方式：依赖管理（Dependency Management）在 dependencyManagement 中声明版本，dependencyManagement 是 Maven 的“版本仲裁中心”，在这里声明的版本会作为项目中该组件的全局统一版本，所有直接/间接依赖都会遵循这个版本。

4.3.标签细节

4.3.1.排除依赖

<exclusions> 用于主动排除依赖包中 “自动传递过来的子依赖”，Maven 依赖默认是 “传递性” 的（比如 A 依赖 B，B 依赖 C，引入 A 会自动引入 B 和 C），在内部可以使用 <exclusion> 来描述需要排除的依赖，这里举个例子：

<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
    <!-- 排除传递依赖 -->
    <exclusions>
        <exclusion>
            <!-- 要排除的子依赖的 groupId -->
            <groupId>org.springframework.cloud</groupId>
            <!-- 要排除的子依赖的 artifactId -->
            <artifactId>spring-cloud-starter-netflix-ribbon</artifactId>
        </exclusion>
    </exclusions>
</dependency>

4.3.2.可选依赖

比如你开发了一个工具包，依赖了 fastjson，但不希望引入你这个工具包的下游项目自动引入 fastjson（避免版本冲突），就给 fastjson 加 <optional>true</optional>。

<dependency>
    <groupId>com.xxx</groupId>
    <artifactId>xxx-common-utils</artifactId>
    <version>1.0.0</version>
    <!-- 标记为可选：引入我的项目的下游项目不会自动引入这个依赖 -->
    <optional>true</optional>
</dependency>

4.3.3.范围依赖

最常用的 scope 值：

  • compile（默认）：编译、测试、运行都生效（如业务核心依赖）
  • test：仅测试阶段生效（如 junit、mockito）
  • provided：编译 / 测试生效，运行时由容器提供（如 servlet-api，Tomcat 已包含）
  • runtime：运行 / 测试生效，编译不生效（如 JDBC 驱动）
<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13.2</version>
    <scope>test</scope>
</dependency>
